## Userspace tests for SEStarnix

These tests run as userspace programs in a minimal Starnix container. The tests
can also run on Linux but the Linux version is not tested in CQ.

### Running the tests

Run on Starnix with:
```
fx test sestarnix_userspace_tests
```

Or use `#meta/<test_name>.cm` to only execute a specific test. E.g.
```
fx test sestarnix_userspace_tests#meta/socketpipes.cm
```

For running the tests on Linux please see [//vendor/google/starnix/tests/selinux/userspace/README.md](../../../../../vendor/google/starnix/tests/selinux/userspace/README.md).


### Audit checker

#### General information

The audit checker collects the audit logs generated by a test and verifies if these
audit logs match the expected ones in `audit_success.json` expectations file.

#### Running tests with audit validation

The audit checker runs automatically when you execute the tests using `fx test sestarnix_userspace_tests`.

#### Updating audit expectations

To generate or update the audit expectations, you must first run the tests on Linux
with the `--update-audit-expectations` flag. This will modify the JSON file containing
the expected audit logs for each test case.

```bash
./run_tests_on_linux.sh --update-audit-expectations
```

After updating, rerun the tests *without* the flag on both Linux and Starnix
to ensure they pass with the new expectations.

#### Audit failures and skipped tests

Sometimes, tests might fail audit checks on Starnix due to expected differences
or issues. You can manage these cases using the following JSON files:

*   `audit_failure.json`: Add entries here for tests that are failing on Starnix
    due to mismatches in audit logs or a different number of audit logs compared
    to Linux.
*   `audit_skip.json`: Use this file to skip audit checks for tests that are
    expected to fail and would affect the audit checker. This can happen,
    for example, if a test manipulates `NETLINK_AUDIT` or if audit logs are
    corrupted on Linux because the tests don't use subprocesses.

#### How the audit checker works

The audit checker is implemented as a gTest listener. For each test case, it:

1.  Sends a "start" sentinel to the audit subsystem.
2.  Runs the test.
3.  Sends an "end" sentinel to the audit subsystem.
4.  Collects all audit logs generated between the start and end sentinels.
5.  Compares these collected logs against the expectations loaded from the JSON file.

A test failure is added if:

*   The audit expectations file cannot be parsed.
*   The generated audit logs do not match the expectations.
*   An expected audit log is missing, or an unexpected denial occurs.
*   A test that should have audit expectations is not included in the expectations file.

The audit checker is a gTest listener that checks the audit logs generated
by each test, to ensure they are the correct permission denials when the
same errno is returned. It sends a start and an end sentinel to the audit
subsystem that will be checked at the end of the test. Every audit log
generated between them will be checked.

Don't forget to run the tests with `--update-audit-expectations` on Linux
first to update the expectations JSON file and then rerun them without it
to ensure the tests produce the correct audit logs. If the audit expectations
are not update, the tests will always fail.

To check that the audit checks are correct:

* If the parsing of the audit expectations file fails, a failure will be added
* If the expectation check fails, a failure will be added
* If an expectation is missed for one of the tests or a setup step produces a denial, a failure will be added
* Make sure all the tests that should have audit expectations are added

If you encounter failures on Starnix, you can use:

* `audit_failure.json` - add here if the test is failing on Starnix because of various reasons such as mismatches in audit logs or different number of audit logs
* `audit_skip.json` - add here if the test is expected to fail because of NETLINK_AUDIT manipulation for example; this can also be used for Linux when audit logs are corrupted because the tests do not use subprocesses


### Writing a test

Add a file to the `tests/` directory, and reference it in the `test_names` list
in BUILD.gn. Don't forget to run the test on Linux before submitting!
