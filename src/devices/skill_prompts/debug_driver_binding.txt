I'm looking to write a skill file that can be used by gemini to help fuchsia driver authors, and AI agents, answer the question "why didn't my driver bind". Below is all the information that the skill should capture and reflect. The skill should go into the file src/devices/skills/debug_driver_binding/SKILL.md. If the skill already exists, ignore it and override the file.

There are a number of things that could lead to this issue, so here are a few reasons this can happen, and what to look for.

The first reason would be that the driver is not included in the fuchsia build. Drivers are included into the build either through a board input bundle, or as part of the platform through an assembly input bundle. If a system is running, this can be checked by using `ffx driver list`, if the driver package is visible in the output, the driver should exist on the device. If its not there, let the user know that the driver is missing from the build.

Next we need to check whether a node matching the bind rules of the driver exists in the driver node topology. There are two types of drivers bind rules, composite and non-composite (we can call these regular). To figure out which type of driver we have, the bind rules need to be checked. The driver's bind rules can be found with the driver code, usually inside of a "meta"/ directory, with the file extension ".bind". Alternatively the bind rules can shown using `ffx driver show fuchsia-pkg://fuchsia.com/xyz#meta/driver.cm`, although the output of this is not as clean as the raw file as it misses symbolizing jump targets, therefore reading the bind file is the preferred approach. If the bind rule starts out with a "composite some_name" or contains multiple nodes or parents, it is a composite. If the bind rules just show up as top level, then it is not composite.


For the regular case we just need to look through the driver nodes, and try to find one where the "Node Properties" matches the driver's bind rules. The bind rules are just a set of rules that decides if the set of key value pairs in the properties match it or not. If a match is found, then the error was likely related to the startup of the driver. The device logs would likely have warnings or errors indicating why it failed to start. Logs can be checked using `ffx log`. If a match is not found, it could be the node properties don't quite match the bind rules, or that it must not have been added by a parent driver at all. To start, first try to run `ffx driver doctor --driver <url>` which will try to find existing unbound nodes and try to find slight misalignments in the rules. If this doesn't find anything, then the second case is more likely. In this case, we would need to investigate whether the parent driver exists, was bound, or why it didn't add the necessary child node. There are a couple of ways to identify which driver would have been the parent and added this node.

If the node needed to declare a fidl capability in its properties, you would need to just search the code base. This would be the case when the bind rules specify a Service capability and transport. For example "Key(fuchsia.hardware.display.engine.Service) == Enum(fuchsia.hardware.display.engine.Service.DriverTransport)", then you would want to look for drivers that provide this service. You can do that by checking the repo for `.cml` files that define a service offer, which is in the form of "capabilities" + "expose" of the service. For example in src/graphics/display/drivers/amlogic-display/meta/amlogic-display.cml is one example of the provider for fuchsia.hardware.display.engine.Service. The DriverTransport value in the enum specifies the transport of the offer, the other transport is ZirconTransport. One check here to make sure the transport is correct is to look for the ".fidl" file and ensure the definition of the protocol and service matches what is here. For example "sdk/fidl/fuchsia.hardware.display.engine/engine.fidl" defines the service as a composition of Engine protocols, and the Engine protocol specifies '@transport("Driver")'. This means DriverTransport is correct. If the annotation is not there, then the default is ZirconTransport. Once you know the transport is correct, move on. There may be multiple drivers that are providers of the same offer, depending on hardware, but usually only one of these is included in the build and the correct one. You can try cross-checking the board input bundles, or checking the driver list as before. Once you have identified the parent, validate that it is included in the build, lets make sure the code is correct. Make sure that you spot in the code where it creates the offer for the service (For example "fuchsia_hardware_display_engine::Service") and provides it to a child node. Ensure that the code is reachable. Namespace renames/aliases might be used in the code (namespace in our example would be "fuchsia_hardware_display_engine"), so you might want to try checking for that as well, both in c++ and rust. If the code looks ok, check to see if it is bound or not. This would be a recursive problem so run through the same workflow this skill describes, just for the parent driver.

Another possibility is that you want a node with a legacy "fuchsia.BIND_PROTOCOL", which is a bit more complicated. You need to figure out the definition for the uint value. For example for "fuchsia.BIND_PROTOCOL == fuchsia.camera.BIND_PROTOCOL.CAMERA_SENSOR_2;" we would want to find the value for CAMERA_SENSOR_2, which you can find by finding all overrides "extend uint fuchsia.BIND_PROTOCOL" in ".bind" files in the repo and finding the enum, for example "extend uint fuchsia.BIND_PROTOCOL {...CAMERA_SENSOR_2 = 0x43,...}". Then you want to find the associated ddk protodef which you can find by looking at "src/lib/ddk/include/lib/ddk/protodefs.h". Here it would be "DDK_PROTOCOL_DEF(CAMERA_SENSOR2,          67, ...)" because 0x43 = 67. This corresponds to the autogenerated C definition ZX_PROTOCOL_CAMERA_SENSOR2, and the banjo protocol ddk::CameraSensor2Protocol. Once you have that you want to check for how its added. For DFv1 parents, this is done through the "proto_id" field on device_add_args, which is either set directly (in our example to ZX_PROTOCOL_CAMERA_SENSOR2), set from the ddktl's set_proto_id (again for the example to ZX_PROTOCOL_CAMERA_SENSOR2), or through having a "ddk::base_protocol" mixin on a banjo protocol that you inherit (in our example that is done by using "ddk::CameraSensor2Protocol<Imx227Device, ddk::base_protocol>"). For DFV2 parents, they would manually create a bind rule using the generated constants "bind_fuchsia::PROTOCOL" and "bind_fuchsia_camera::BIND_PROTOCOL_CAMERA_SENSOR_2". You need to look for both types, and once again identify which one is the correct one based on which one is available in the build. And once you have, the same recursive problem exists, run through this skill with that driver.

Another possibility is that you have a platform device, aka "pdev". These are added by the platform bus driver. The platform bus driver is either using a devicetree to know which platform devices to add, or the platform bus driver is hardware specific and manually creates everything that is required. The way to find the platform device is using its PID/VID/DID. For devicetree based boards, you need to find the visitor that is responsible for adding the specific node. You can find all visitors by looking for C++ inheritors of "fdf_devicetree::Visitor".

If none of the above criteria matches the node you're looking for, just exit and let the user know the properties that you do see it needs. Ask the user if they are able to locate the would-be parent themeselves. If they do identify it and give it to you, then you can proceed by recursing and investigating that using the same skill.


For composite drivers we have a list of things that we need to check. Composite drivers bind to multiple parents, but not directly, they bind to a completed composite node that is parented by multiple parent nodes. There is a piece of glue that is called the composite parent spec(ification), sometimes also called the composite node spec. This composite parent spec, contains a list of parents, each with two sides. One side is the bind rules, these match up to the properties of nodes in the node topology, very similar to the non-composite bind rules. The other side is the properties side. These match up to the composite bind rules of the driver. The spec is added through the board driver generally, which is either a manually written piece of code, or done as part of a devicetree board driver through more generic visitors that parse a devicetree. Another way to think of the spec is that its a translation layer between hardware specific properties, to more generic properties that a driver can specify in its bind rules. Therefore both sides of the spec have to be satisfied in order for the completed composite node to appear and have the composite driver bind to it. To start, run the `ffx driver doctor --driver <url>` command. This will show any composite specs that the driver has matched with (that means the driver to spec side is satisfied), as well as try to look through any specs that don't have any matches and try to find slight discrepancies that could be causing the issue. If there are no matched specs to the driver, then that means the board driver did not add all of the necessary (non-optional) parents into the composite spec definition. All non-optional nodes need to be present in the spec and matched for success. The output will also print out a command with successfully matched specs, to investigate the spec further. This will look like `ffx driver doctor --composite-node-spec <name>`. The output of this command will show the other side of the spec, which is whether each parent was able to be matched to a node in the topology. If any of the nodes are "Unbound", that means that the node side of the spec is not satisfied. The next step if you encounter an Unbound parent, is to look at its Bind Rules, which you can do using `ffx driver composite show <name>`, because those rules did not match a node in the topology. The steps here are exactly the same as they would be for a non-composite driver that would have those same bind rules. So you would try to identify the parent driver that would add a node matching those bind rules in the missing parent, or why an existing node doesn't quite line up with it. You can do this recursively by running through this skill again (skipping the part that ensures the driver is included in the build, as you aren't looking for a driver, just the node).


At each of these steps you should tell the user what you are looking for and why.

If any of the referred to `ffx` commands cannot be run (for example if it timed out), the same commands can be ran over a serial connection to the device, just without the `ffx` part. From then on in the session use serial. So for example `ffx driver list` would become `driver list` when run over serial. To run serial commands, use the run_on_serial skill.

More info about composite specs can be found at docs/contribute/governance/rfcs/0197_node_groups.md, and docs/development/drivers/developer_guide/create-a-composite-node.md.

Some other helpful commands not mentioned above that can help if stuck:
- `ffx driver composite list -v` lists all composite specs and a driver match if exists
- `ffx driver node list` lists all nodes in the topology
- `ffx driver node show <name>` shows the properties of a specific node
